## Project Overview

This course simulates a real-world software engineering team environment — applying Agile/Scrum methodology, collaborative Git workflows, and structured project management to deliver a working software product as a cross-functional team. It is the most professionally realistic course in the WGU CS program, because it replicates what every software engineer experiences from day one on the job.

## What Was Built

A collaborative software project developed across multiple sprints using Agile principles. The team applied:

### Project Lifecycle Phases

**Phase 1 — Project Planning & Discovery**
- Defined project scope and requirements collaboratively as a team
- Wrote user stories in the format: *"As a [user], I want to [do something] so that [I get value]"*
- Estimated effort using story points
- Created and prioritized the product backlog
- Defined the Definition of Done (DoD) for all work items

**Phase 2 — Sprint Execution (Iterative Development)**
- Ran structured 2-week sprints with daily standups
- Each sprint started with Sprint Planning and ended with Sprint Review + Retrospective
- Team members picked up backlog items, created feature branches, wrote code, and submitted pull requests
- Code reviewed by team members before merge — no direct commits to `main`

**Phase 3 — Integration and Testing**
- Integrated feature branches from multiple team members into a working build
- Resolved merge conflicts collaboratively
- Performed integration testing to ensure components worked together
- Documented defects and tracked resolution through the sprint board

**Phase 4 — Delivery and Documentation**
- Delivered a working software increment at the end of each sprint
- Wrote project documentation including architecture overview, setup instructions, and user guide
- Conducted a final Sprint Review demonstrating the complete product

## Agile / Scrum Practices Applied

| Practice | How It Was Applied |
|---|---|
| **Product Backlog** | All features and tasks tracked as user stories, prioritized by value |
| **Sprint Planning** | Team committed to a realistic sprint goal each iteration |
| **Daily Standups** | Brief synchronization — what did I do, what will I do, any blockers |
| **Sprint Review** | Working software demonstrated to stakeholders at end of each sprint |
| **Sprint Retrospective** | Team reflection — what went well, what to improve, action items |
| **Burndown Charts** | Visualized remaining work to track sprint progress |
| **Definition of Done** | Clear, agreed criteria before any item was marked complete |

---

## Collaborative Git Workflow

| Practice | Implementation |
|---|---|
| Branch-per-feature | Every user story had its own feature branch |
| Pull requests | No code merged to `main` without a PR and peer review |
| Merge conflict resolution | Conflicts resolved collaboratively with team context |
| Commit conventions | Descriptive commit messages tied to backlog item IDs |
| Protected main branch | Main branch locked — only merged via approved PRs |

## Team Dynamics and Soft Skills

This course specifically developed the human side of software engineering — the skills that determine whether an engineer is effective on a team:

- **Written communication** — clear PR descriptions, commit messages, and async updates that teammates could act on without a meeting
- **Giving and receiving code review** — providing constructive feedback on others' code, incorporating feedback on my own without ego
- **Conflict resolution** — resolving technical disagreements about implementation approaches with data and reasoning, not preference
- **Scope management** — recognizing when a feature was growing beyond the sprint capacity and communicating that proactively
- **Dependency management** — coordinating with teammates whose work mine depended on, unblocking each other

## Why This Matters to a Hiring Manager

Most candidates can write code in isolation. What companies actually need is engineers who can work within a team to deliver software predictably. This course proves I can:

1. **Operate inside a Scrum team** — standups, sprint planning, retrospectives are not new to me
2. **Use Git collaboratively** — branching, PRs, code review, conflict resolution on shared code
3. **Manage my own work** — pick up backlog items, estimate them accurately, and deliver on commitments
4. **Communicate proactively** — raise blockers early, write clear status updates, keep the team informed
5. **Iterate and improve** — retrospectives taught me to reflect on process, not just output

These are the skills that determine whether someone is a productive team member from week one — not in month six after they've figured out "how things work here."
